COORD v0.1 — Coordination Protocol for Autonomous Agents

The HTTP of the agent stack. Framework-agnostic task definition, lifecycle, state handoff, dependency resolution, escalation, and human-in-the-loop.

L4 Application CrewAI / LangGraph / Claude SDK
↑ COORD ← task semantics, lifecycle, state handoff
L3 Communication A2A (transport) + MCP (tools) + MCP-DS (discovery)
↑ MCP-DS
L2 Runtime LangGraph / OpenAI SDK / Smolagents
↑ OpenAI API format
L1 Models GPT-4o / Claude / Gemini / vLLM

A2A transports messages between agents (like TCP). COORD defines what those messages mean (like HTTP). Frameworks that implement COORD interoperate for free — a CrewAI agent can hand off to a LangGraph agent, which delegates to a Claude SDK sub-agent, all speaking the same coordination language.

1. Core Model

Task — the atomic unit of work

A task is a sealed envelope. Any agent that understands COORD can accept it, process it, and return it.

{
  "spec_version": "0.1",
  "task_id": "tsk_01j5c8x9y0",
  "goal": "Research competitor pricing",
  "meta": {
    "trace_id": "trc_abc123",
    "workflow_id": "wf_001",
    "created_at": "2026-06-09T21:00:00Z"
  },
  "inputs": {
    "companies": ["OpenAI", "Anthropic"],
    "detail_level": "pricing_tiers"
  },
  "dependencies": {
    "parent": "tsk_01j5c8x9wz",
    "blocked_by": [],
    "priority": "normal"
  },
  "state": {
    "pointer": "memory://wf_001/state",
    "scope": "workflow"
  },
  "constraints": {
    "deadline": "2026-06-10T10:00:00Z",
    "max_tokens": 50000,
    "allowed_tools": ["web_search", "filesystem_read"],
    "budget_max_usd": 0.50
  },
  "output_contract": {
    "format": "markdown",
    "schema_ref": "https://schemas.example.com/research-report.json",
    "required": ["summary", "table", "sources"]
  },
  "escalation": {
    "on": ["timeout", "error", "uncertainty"],
    "to": "agent:coordinator::workflow_001",
    "auto_retry": 2
  }
}

2. Coordination Lifecycle

The lifecycle every agent agrees on:

                      ┌──────────────────────┐
                      │     CANCELLED         │
                      └──────────────────────┘
                                ▲
                                │
┌──────────┐    ┌──────────┐   │   ┌──────────┐    ┌──────────┐
│ CREATED  │───▶│ ASSIGNED │───┴──▶│IN_FLIGHT │───▶│COMPLETED │
└──────────┘    └──────────┘       └──────────┘    └──────────┘
                                       │  │              │
                                       │  ▼              │
                                       │  ┌──────────┐   │
                                       ├─▶│NEED_HELP │───┤
                                       │  └──────────┘   │
                                       │                 │
                                       ▼                 ▼
                                  ┌──────────┐      ┌──────────┐
                                  │  FAILED  │      │  REVIEW  │─▶ COMPLETED
                                  └──────────┘      └──────────┘
                                       │                  │
                                       ▼                  ▼
                                  ┌──────────┐      ┌──────────┐
                                  │ESCALATED │      │ REJECTED │
                                  └──────────┘      └──────────┘
                                       │
                                       ▼
                                  ┌──────────┐
                                  │REASSIGNED│──▶ IN_FLIGHT
                                  └──────────┘
StateMeaning
CREATEDTask exists, not yet assigned
ASSIGNEDBound to a specific agent
IN_FLIGHTAgent is actively working
NEED_HELPAgent hit uncertainty, requesting guidance
REVIEWWork complete, awaiting human/coordinator approval
COMPLETEDSuccessfully finished
FAILEDIrrecoverable error
ESCALATEDEscalated to parent coordinator
CANCELLEDAborted (user or parent coordinator)
DELEGATEDPassed to a sub-agent (hierarchical)
Key insight: State transitions are the protocol constraint. An agent receiving a task in CREATED knows exactly what to do. An agent receiving COMPLETED knows not to touch it. This is the TCP state machine for agent coordination.

3. State Handoff Protocol

The most expensive mistake in current multi-agent systems: copying the entire context between agents. COORD uses durable state pointers instead.

state.pointer → a URI that resolves to durable state
state.scope   → how visible the state is

State Scopes

ScopeVisibilityExample
taskThis task onlyIntermediate results
workflowAll agents in this workflowShared research notes
user_sessionAcross workflows for a userUser preferences
globalAll agents (careful!)Registered capabilities

State Operations

// Read
{ "state_op": "read", "pointer": "memory://wf_001/state", "version": 3 }

// Write  
{ "state_op": "write", "pointer": "memory://wf_001/research_openai",
  "value": { "pricing": "$0.15/1M input", "models": ["GPT-4o", "o3"] },
  "scope": "workflow" }

Why pointers beat history-passing: Agent A writes a finding to memory://wf_001/research_openai. Agent B reads from the same pointer. No shared context on A's side. No redundant token passing. You get git-like versioning — roll back to "before Agent D wrote to state."

4. Coordination Graph

A workflow is a DAG of tasks:

{
  "workflow_id": "wf_001",
  "goal": "Generate competitive analysis report",
  "tasks": [
    { "task_id": "tsk_01", "goal": "Research OpenAI pricing" },
    { "task_id": "tsk_02", "goal": "Research Anthropic pricing" },
    { "task_id": "tsk_03", "goal": "Research Google pricing" },
    { "task_id": "tsk_04", "goal": "Write report",
      "depends_on": ["tsk_01", "tsk_02", "tsk_03"] },
    { "task_id": "tsk_05", "goal": "Review and approve",
      "depends_on": ["tsk_04"], "hitl": true }
  ],
  "state": {
    "pointer": "memory://wf_001/state",
    "shared_keys": {
      "research_*": "any agent can read",
      "report_draft": "only writer + reviewer"
    }
  }
}

The coordinator resolves dependencies — tsk_01, tsk_02, tsk_03 run in parallel. tsk_04 waits for all three. tsk_05 requires human approval.

5. Human-in-the-Loop

A standard format for any agent to request human intervention:

{
  "task_id": "tsk_05",
  "state": "NEED_HELP",
  "hitl_request": {
    "type": "approval",
    "message": "The draft report recommends lowering prices by 15%. Approve?",
    "context": {
      "summary": "OpenAI $0.15/1M, Anthropic $0.12/1M...",
      "recommendation": "Drop to $0.10/1M to undercut",
      "tradeoffs": "Margin drops 60%→55%, volume estimated +40%"
    },
    "options": [
      { "id": "approve", "label": "Approve recommendation" },
      { "id": "modify", "label": "Suggest different price", "input_required": true },
      { "id": "reject", "label": "Keep current pricing" }
    ],
    "timeout_s": 86400
  }
}

HITL is just another state transition: IN_FLIGHT → NEED_HELP. The coordinator routes it to whatever human interface is available. The agent doesn't care if it's Slack, email, or a web dashboard.

6. Error Handling & Escalation

Error TypeMeaningDefault Action
timeoutExceeded max durationRetry once, then escalate
tool_failureMCP tool returned errorRetry with backoff
uncertaintyAgent lacks confidenceEscalate with NEED_HELP
invalid_inputTask doesn't match capabilitiesFail immediately
resource_exhaustedHit rate/token/budget limitsEscalate
state_conflictVersion conflict on state writeRetry with newer version
coordinator_unreachableCan't report stateBuffer and retry

Escalation always goes back to the coordinator, which decides: retry, reassign, skip, or cancel.

7. Transport Mapping

COORD is transport-agnostic. Default recommendation: COORD over A2A.

TransportHow COORD rides on it
A2ACOORD task envelope wraps A2A Task.message. State pointer maps to A2A state field.
HTTPPOST /coord/v1/tasks with COORD envelope
gRPCrpc SubmitTask(CoordTask) returns (CoordStatus)
MCPExposed as a coord_execute tool resource

8. Framework Interop

The same workflow definition can run on any framework:

workflow:
  id: wf_001
  goal: "Weekly competitive analysis"
  tasks:
    researcher:
      goal: "Research competitors"
      accepts_pattern: "Research *"
      tools: [web_search, filesystem]
    writer:
      goal: "Draft report"
      accepts_pattern: "Draft *report*"
      depends_on: [researcher]
    reviewer:
      goal: "Approve report"
      depends_on: [writer]
      hitl: true

This maps naturally to:

  • CrewAI — researcher/writer/reviewer as role-based agents
  • LangGraph — nodes with state edges
  • Claude SDK — sequential agent calls
  • Custom — direct function calls with coordinator loop

The task envelope, state format, and lifecycle are identical across all of them. Only the internal routing differs.
